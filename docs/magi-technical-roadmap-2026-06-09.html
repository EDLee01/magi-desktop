Magi 技术路线与稳定性保护基线

本文档是 Magi 从 v0.1.8 向可靠 coding agent 演进的执行依据。 它不是愿望清单，而是版本边界、问题处置方案、验收门禁、兼容策略和回滚方法的统一技术基线。 后续迭代必须优先保证现有稳定能力不回退。

1. 核心决策

第一目标： Magi 必须成为一个用户敢于交付真实代码任务、能够持续执行、能够被审阅、能够停止和恢复的 coding agent。 对齐 Codex 体验时，优先对齐执行可信度和工作流闭环，而不是表面功能数量。

必须优先完成

  • OS 级执行沙箱与精确审批边界。
  • 长任务、PTY、进程树、取消和恢复。
  • Diff、Review、逐 hunk 接受/撤销与回滚。
  • 稳定 TUI、会话恢复、上下文与后台任务可见性。
  • Worktree 隔离和有边界的 Subagent 协作。
  • MCP、Hooks、Memory 的生命周期可靠性。

明确不进入近期范围

  • Computer Use：不做通用桌面、浏览器点击和 OS GUI 自动化对齐。
  • 不为了“工具数量”继续添加低价值工具。
  • 不做大规模重写，不替换已稳定的 TypeScript 核心。
  • 移动端和远程控制只在 coding agent 核心稳定后继续增强。
  • 不到真实可用和可回滚，不使用 1.0.0 版本号。
允许的行为变化： 安全缺陷修复可以有意收紧旧行为，例如 acceptEdits 不再放行非编辑类副作用工具。 这种变化必须写入兼容性说明、增加回归测试，并提供明确替代方式；不能把安全收紧伪装成“完全无行为变化”。

4. 当前问题与可执行处置

P0 安全与执行边界

ID 问题与证据 影响 实施方案 验收
SEC-01 src/tools/disk-usage.ts:34 使用字符串 execSync 拼接 path； src/tools/which.ts:23 对 shell 参数做不完整转义。 被标记为只读工具但可形成命令注入，绕过审批语义。 统一改为 spawnSync(executable, argv)；禁止 shell； path/name 使用 schema 限制；建立 no-shell-interpolation 静态检查； 审计所有字符串型 execSync。 注入 payload 不能执行；工具仍返回原有业务结果；新增 Linux/macOS 测试； 扫描不再发现用户输入进入 shell 字符串。
SEC-02 src/tools/registry.ts:774 对 acceptEdits 直接放行全部工具。 “接受编辑”可能放行非编辑副作用，例如进程、网络、配置或远程操作。 将工具风险拆成 read、workspace_edit、command、 network、external_write、destructive； acceptEdits 仅自动允许 workspace edit； 其他类别继续 ask/deny。 权限矩阵逐工具覆盖；旧 FileWrite/Edit/Patch 流程不增加额外打断； 网络、命令和远端写入不能借 acceptEdits 自动通过。
SEC-03 Rust runner 存在，但当前未形成强制 OS 级沙箱边界。 审批依赖应用层判断；模型生成命令仍可能访问过宽的文件系统或网络。 Linux 使用 bubblewrap；macOS 使用 sandbox-exec/Seatbelt 可行路径； 工作区可写、.git/.magi-next 受保护、网络默认关闭； 命令、文件编辑和子进程统一经过 runner policy。 越界写、网络、Unix socket、私网地址均有否定测试； 沙箱不可用时 fail closed 或明确降级并要求授权，不能静默全权限。
SEC-04 magi kill 调 Control API 时未附带 daemon 鉴权。 启用鉴权后命令可能 401，取消能力失效。 daemon 状态文件保存 token reference，不保存明文到日志； CLI 统一走 Control API client，自动加载 token； token 文件权限固定为 0600。 鉴权开启时 magi kill 成功；无 token 明确报错；日志不泄漏 token。

P2 依赖、CI 与产品债务

  • Node 版本：package.json 仍声明 >=20，CI 仍测试 Node 20； 2026-06-09 的实际安装基线已使用 Node 22。下一版本统一到 Node 22 LTS，停止承诺 Node 20。
  • Vitest：锁定的 3.2.4 有开发服务器安全公告； 在不跨大版本的前提下升级到已修复版本，并全量验证。
  • Playwright 预检：npm 包存在不代表浏览器二进制存在； verify 前增加 playwright install --dry-run 或明确的 doctor 检查。
  • MCP：需要更强的 reconnect/backoff、resource list change、roots 和 OAuth 生命周期。
  • Memory：当前关键词和图召回有效，但还缺自动生成的秘密清理、老化策略和跨项目策略边界。

5. 目标技术架构

保留 TypeScript 产品层，强化 Rust runner 为执行安全边界。禁止为了技术偏好进行 Rust 全量重写。 

┌──────────────────────────────────────────────────────────┐
│ Magi Product Layer (TypeScript / Node 22 LTS)           │
│ CLI · TUI · Session · Goal/Plan · Memory · MCP · Skills │
│ Provider Router · Review · Control API · Audit          │
└───────────────────────┬──────────────────────────────────┘
                        │ typed JSON-RPC / event protocol
┌───────────────────────▼──────────────────────────────────┐
│ Magi Runner (Rust)                                      │
│ PTY · process groups · timeout · cancel · patch apply   │
│ filesystem policy · network policy · sandbox · audit    │
└───────────────────────┬──────────────────────────────────┘
                        │ OS-enforced boundary
┌───────────────────────▼──────────────────────────────────┐
│ Linux bubblewrap / macOS sandbox / Windows Job Object   │
│ workspace roots · protected paths · network deny-first  │
└──────────────────────────────────────────────────────────┘

Shared durable state:
SQLite session/job/audit/memory metadata + append-only event records

Execution surfaces:
Local TUI · headless CLI · web panel · remote control client

架构原则

  1. 一个执行入口：命令、PTY、后台任务、Subagent 命令最终统一进入 runner。
  2. 两个独立控制：Sandbox 决定“技术上能做什么”，Approval 决定“何时询问”。
  3. 事件先于 UI：所有状态先写结构化事件，再由 TUI、CLI、Panel 渲染，避免多套状态逻辑。
  4. Git 作为变更真相：Review、stage、revert、worktree 都基于 Git，不创建第二套版本系统。
  5. 兼容优先：CLI/JSON/SQLite 变化必须有 schema version 和 migration，不做隐式破坏。
  6. 失败关闭：权限、沙箱、token、迁移状态不确定时拒绝执行并给出恢复路径。

6. 不让稳定功能变得不稳定

发布红线： 只要旧能力合同中有一项下降、完整 verify 未通过、迁移无法回滚、或已知 P0 未关闭， 该版本不得发布，也不得把版本号标记为完成。

6.1 变更隔离

  • 每个版本使用独立分支和 Git worktree，不直接在稳定 main 上开发。
  • 一个版本只处理该版本问题清单，禁止顺手重构无关模块。
  • 安全修复、runner、TUI、Subagent 不在同一版本混改，降低定位难度。
  • 新实现先通过 feature flag 或新入口接入；旧路径保留到新路径连续通过门禁。

6.2 测试保护

  • 现有 672 个测试是最低集合，不允许通过删除、skip、放宽 timeout 或降低断言来“修复”失败。
  • 每个 bug 先写失败测试，再修代码；安全问题必须有恶意输入测试。
  • 每个版本增加专属 regression suite，并纳入 npm run verify。
  • 高风险模块采用 mutation/fault injection：权限矩阵、进程取消、SQLite 迁移、SSE reconnect。

6.3 数据保护

  • 所有数据库迁移先备份，再在事务中执行；失败自动恢复旧文件。
  • 迁移只允许 additive-first：先加字段/表，再双读，最后才移除旧字段。
  • 配置新增字段必须有默认值；未知字段不能被旧版本静默删除。
  • 跨版本回滚必须验证：新版本产生的数据，至少能被旧版本安全忽略或由 downgrade 工具转换。

6.4 性能与体验保护

  • 记录启动时间、首 token 时间、工具调度延迟、内存占用和完整任务耗时。
  • 默认核心工具数量和 system prompt 大小设预算，新增能力不能无上限扩大上下文。
  • TUI 新实现必须保留无颜色、窄终端、非 TTY 和机器输出路径。

6.5 允许修改稳定行为的唯一流程

  1. 建立 Compatibility Delta，明确旧行为、风险和新行为。
  2. 提供替代命令、配置或迁移步骤。
  3. 增加旧行为拒绝测试和新行为成功测试。
  4. 在 release notes 中单独列为 Breaking/Security Change。
  5. 用户未确认前，不修改与当前版本范围无关的默认值。

7. 版本化技术路线

工时按 AI 实际执行小时拆分。每个版本独立验收、独立回滚；上一个版本未通过，不开始下一个版本的写入工作。

v0.1.9 · Security & Baseline Hardening H1-H20

目标：关闭已确认安全/正确性问题，统一 Node 22 基线，建立后续版本不可回归合同。

  • H1-H2：冻结 v0.1.8 capability manifest、输出 schema fixtures、回滚清单。
  • H3-H5：Node engine/CI 升到 22；升级 Vitest 修复安全公告；增加 Playwright preflight。
  • H6-H9：修复 DiskUsage/Which 注入；审计所有 shell 字符串；增加静态规则。
  • H10-H13：实现风险分类权限矩阵；收紧 acceptEdits；保留 FileWrite/Edit/Patch 低打扰体验。
  • H14-H17：修复 SSH、Panel cwd、文件 mode、magi kill 鉴权。
  • H18-H20：全量 verify、升级/回滚 smoke、release notes、归档报告。

退出条件：P0 全部关闭；critical audit 为 0；672 旧测试全通过；8/8 能力门禁 1.00；无 CLI/JSON 兼容回退。

v0.2.0 · Runner, Sandbox & Durable Jobs H1-H28

目标：把执行权限从应用层约定提升为 OS 强制边界，并完成可靠取消与恢复。

  • H1-H4：定义 runner protocol v1、能力协商、错误码和审计事件。
  • H5-H10：Linux bubblewrap workspace-write/read-only/full-access；保护 .git 和状态目录。
  • H11-H14：网络默认 deny；域名 allowlist；localhost/私网/Unix socket 独立策略。
  • H15-H19：统一 PTY/命令执行；process group；超时和取消；stdout/stderr 背压。
  • H20-H23：durable job state；daemon 重启后 interrupted/recoverable 状态。
  • H24-H28：故障注入、跨平台降级、性能基线、完整发布门禁。

退出条件：无 runner 绕行路径；越界写和网络请求被 OS 拒绝；取消后无残留子进程；重启状态可解释、可恢复。

v0.2.1 · Review, Diff & Rollback H1-H20

目标：形成“Agent 修改后，用户能精确审阅、部分接受和可靠回退”的闭环。

  • H1-H4：Review scope：uncommitted、commit、base branch、custom instructions。
  • H5-H8：独立 reviewer session；findings 严重度、文件和行号合同。
  • H9-H12：last-turn diff 和 turn snapshot；文件/hunk stage、unstage、revert。
  • H13-H16：inline feedback event model；CLI/TUI 共用 review 数据。
  • H17-H20：脏工作树、rename、二进制、冲突和回滚 E2E。

退出条件：/review 不再只是提示；用户可只保留部分变更；revert 不覆盖进入会话前的用户改动。

v0.3.0 · TUI v2 H1-H28

目标：在不破坏 headless/JSON 的前提下，达到稳定、可观察、键盘优先的交互体验。

  • H1-H5：基于事件流的 transcript renderer，保留 native scrollback 策略。
  • H6-H10：流式 Markdown、语法高亮、diff、工具生命周期折叠。
  • H11-H15：状态栏：model、permission、context、tokens、git、goal、background jobs。
  • H16-H19：queued follow-up、Ctrl+R、file mention、prompt editor、图片占位。
  • H20-H23：审批、AskUserQuestion、Review、Memory、Agent thread overlay。
  • H24-H28：PTY snapshots、窄终端、无颜色、Unicode、性能和旧 TUI fallback。

退出条件：新 TUI 连续两轮完整门禁通过后才成为默认；旧 TUI 至少保留一个版本作为 fallback。

v0.3.1 · Worktree & Subagent Coordination H1-H24

目标：把已有真实 Subagent 扩展为可审阅、可隔离、不会互相破坏的并行工作流。

  • H1-H4：Agent thread 状态和 /agent 切换；主线程只接收摘要。
  • H5-H9：写 Agent 默认独立 worktree；读 Agent 允许共享只读 workspace。
  • H10-H13：write claim 从声明升级为执行期强制；冲突自动停止。
  • H14-H17：Agent result diff、review、merge/handoff；冲突不得自动覆盖。
  • H18-H21：worktree snapshot、磁盘配额、清理和恢复。
  • H22-H24：并行探索、并行验证、写冲突和 daemon 重启 E2E。

退出条件：两个写 Agent 不能修改同一文件；worktree 删除前有 snapshot；主工作树用户改动不受影响。

v0.4.0 · MCP, Hooks & Memory Reliability H1-H20

目标：让扩展生态在断线、变更和不可信输入下仍可预测。

  • H1-H4：MCP reconnect/backoff、session expiry、resource list changes、roots。
  • H5-H8：OAuth token lifecycle、最小权限存储、明确的 auth-required 恢复。
  • H9-H12：Hook hash trust、变更后重新审批、timeout/cancel、来源可见性。
  • H13-H16：Memory secret redaction、生成开关、老化和跨项目边界。
  • H17-H20：不可信 MCP/Hook 故障注入、恢复测试、审计报告。

退出条件：扩展失败不拖垮主 Agent；变更后的 hook 不会静默运行；Memory 不保存测试中可识别的秘密。

v0.5.0 · Productization & Remote Control H1-H16

目标：在核心稳定后完善安装、升级、诊断和远程 coding 控制，不扩展为 Computer Use。

  • H1-H4：安装/升级/降级、shell completion、doctor fix suggestions。
  • H5-H8：Control API version negotiation、remote auth、TLS/SSH forwarding 文档。
  • H9-H12：Web/mobile control 聚焦 session、审批、任务和 review，不执行设备本地 shell。
  • H13-H16：发布包、迁移演练、支持矩阵、用户文档和 telemetry-off 验证。

退出条件：新机安装、旧状态升级、降级和远程控制都有自动化 smoke；不包含桌面 GUI 自动化。

1.0.0 准入条件

  • 至少三个连续版本没有 P0/P1 数据丢失或权限绕过回归。
  • Linux/macOS 主支持平台完整门禁稳定，Windows 有明确支持或明确实验状态。
  • 升级和降级演练通过；状态备份恢复有自动化测试。
  • 真实 provider coding smoke 可重复，不依赖 mock 结果宣称可用。
  • 默认权限模式具备 OS 级沙箱；无静默 full access。
  • Review/rollback、长任务取消恢复、worktree agent workflow 达到可日常使用。

8.2 分层门禁

层级 验证内容 失败处理
G0 环境 Node/Rust/Git/rg/Playwright 浏览器/沙箱能力/磁盘空间。 停止测试，报告环境缺口；不能把环境失败解释为代码失败。
G1 静态 Format、lint、type、build、禁止危险 API 扫描。 修复后从 G1 重跑。
G2 合同 672+ 旧测试、CLI/JSON fixtures、SQLite migration tests。 任何旧合同失败即阻断。
G3 能力 8/8 capability、black-box、complex harness、model tasks。 不得降低阈值；定位具体能力再修。
G4 安全 注入、越界写、网络、token、symlink、process tree、秘密扫描。 P0/P1 安全失败禁止发布。
G5 真实环境 TTY、浏览器 Panel、daemon restart、真实 provider opt-in smoke。 标明 provider 外部故障和产品故障，不能用 mock 代替真实 smoke。
G6 迁移回滚 旧状态升级、新状态读取、失败恢复、版本降级。 无法回滚即不得发布。

8.3 CI 调整

  • CI 主矩阵改为 Node 22；Node 24 作为前瞻 allow-failure，稳定后再纳入 required。
  • Linux 负责完整 capability/Playwright/sandbox；macOS 负责 CLI、TTY、runner 和基础 sandbox。
  • 每个 PR 上传 capability JSON、测试摘要、license、SBOM 和失败 artifact。
  • Nightly 增加重复执行，检测 flaky rate；单次偶然通过不能掩盖不稳定。
  • 对 LSP、browser 和本地端口测试设置环境 preflight，不通过时给出明确 blocker。

9. 数据、配置与协议兼容策略

9.1 SQLite

  1. 新增 schema_migrations 表，记录 version、checksum、applied_at。
  2. 迁移前复制数据库到 ~/.magi-next/backups/<timestamp>/。
  3. 迁移在单事务中执行；失败 rollback，并保留诊断日志。
  4. 新版本首次启动后先运行只读检查，再执行 migration。
  5. 破坏性 schema 删除至少延迟两个 minor 版本。

9.2 配置

  • 配置结构增加显式 configVersion。
  • 未知字段 round-trip 保留，避免旧版本覆盖新字段。
  • 默认值变化必须是 Compatibility Delta，并提供 doctor 输出。
  • 秘密只从 env 或 0600 secret store 读取，不写入普通 YAML、日志和报告。

9.3 CLI 与事件协议

  • text 输出可优化排版，但 JSON 和 stream-json 字段必须 versioned。
  • 新增字段只做 additive；删除/重命名字段需保留兼容 alias。
  • Control API 提供 /version 和 protocol negotiation。
  • runner JSON-RPC 明确 protocol version，TS 与 Rust 不允许各自猜测。

10. 每个版本的开发、交付与回滚流程

  1. 基线冻结： 记录起始 commit、工作树状态、Node/Rust 版本和完整 verify 结果。
  2. 建立隔离 worktree： 分支只包含当前版本范围；禁止在 main 直接实现。
  3. 先测试后修复： 每个问题建立复现和验收测试，再修改实现。
  4. 小批提交： 安全、迁移、TUI、文档分别提交，确保可二分和可单独回退。
  5. 完整门禁： 先 focused tests，再 unit，再完整 verify，再真实 smoke。
  6. 产物归档： source/runtime archive、SHA-256、SBOM、license、capability report、release notes。
  7. 回滚演练： 安装新版本、创建状态、降级到旧版本、恢复备份并运行 doctor。
  8. 发布后观察： 保留上一版本命令和归档；出现 P0 立即停止 rollout。
