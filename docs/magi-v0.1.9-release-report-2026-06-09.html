发布结论

关键修改

领域问题v0.1.9 处理
运行时Node.js 20 已不适合作为新版本基线最低版本提升至 Node.js 22，CI 增加 Node.js 24 实验矩阵和启动前检查。
命令安全部分系统工具使用 shell 字符串拼接改为 argv 调用或 JavaScript 过滤，并加入命令注入回归测试。
权限acceptEdits 边界过宽只自动允许三种工作区编辑工具；命令、网络、远程、状态变更和破坏性操作仍需明确授权。
SSH路径引用、Unicode 字节数和写入传输存在正确性风险使用 POSIX 数据引用和 stdin base64 写入，增加模拟测试与真实 OpenSSH 验证。
Control API任务 cwd 可能脱离 daemon 工作区对 realpath、存在性和工作区边界做服务端强制校验，拒绝符号链接逃逸。
Daemon本地 kill 流程缺少可靠认证增加受文件权限保护的本地凭据和认证重试。
文件写入原子替换可能改变原文件 mode默认继承现有 mode，除非调用方明确指定。

兼容性变化

  • Node.js 最低要求从 20 提升到 22。
  • 依赖 acceptEdits 自动执行 shell 的脚本必须增加范围明确的 allow 规则。
  • Control API 的 cwd 必须是 daemon 工作区内已经存在的真实目录。
  • 状态格式无需迁移，升级与回滚已做自动化实证。

已知限制和后续入口

下列内容没有被包装为“已完成”。后续版本必须在不降低本报告验证指标的前提下逐项推进。
  • TUI 仍是 partial，缺少完整 transcript 级回归覆盖。
  • subagent 写操作尚未强制 worktree 隔离。
  • Control API 尚未完成两台独立机器之间的端到端验证。
  • Rust runner 尚未实施操作系统级 sandbox。
  • Memory、Control API、subagents 和 Rust runner 仍标记为 beta。
  • Computer Use 明确排除在 Magi 产品边界外。

稳定性保护规则

  1. 每个版本先冻结可回滚基线，再修改共享行为。
  2. 权限、文件写入、进程、远程连接和状态格式变更必须有定向回归测试。
  3. 模拟评测只能用于可重复回归，真实业务结论必须有真实 provider 或真实协议证据。
  4. 发布必须保留源码、npm 包、报告、SBOM、许可证清单和 SHA-256 校验文件。
  5. 任何核心指标低于本版本基线时不得发布。
